Real-know-Git

Aug 01, 2018 in common

你真的了解Git嘛，还是只是会用git clone,git push?
曾经我以为，git只是个简单的硬盘，并不会对实际工作产生什么帮助
不过最近与大佬一同工作，受到很大启发 This is Git!

GitFlow

首先按照Git的标准工作流程，必须要有两个分支，master/dev
开发人员是不去碰master，只有当测试工作完成，由项目负责人审核通过，项目负责人把dev分支合并到master分支。

按照git flow的工作流程，项目总共分三种迭代方式。

  • 新功能
  • bug修改
  • 发行版更新

当你来了一个业务需求，比如cv功能，就要从dev分支新建一个新功能feature分支，所有更改在feature分支完成，push到远程feature分支，测试完成后，与dev分支合并，供测试人员测试。测试完成后，由负责人合并到master分支，申请发布时间，发布上线。

如果你的线上版本出现问题，需要紧急修复，就从master分支新建一个bugfix分支，进行修复，完成后同时同步到master和dev分支。保证dev分支要至少与master同步，通常的情况，尽量保证dev分支版本高于master分支，

但是我们在开发过程中，有一种情况比较棘手，这种情况是，比如一个实验性的改动，或者重构，可能改动了大量的代码，这个工作并不是短期的工作，可能需要2个月时间，但这个期间接到了其他任务，比如你的项目有个bug需要修改，那这种情况，你很可能忘记更改分支，因为写新东西和富有挑战的任务总是会让人兴奋。这时候你只是提交了一次，并没有更改分支，继续在测试分支更改老版本的bug。
当你写到一般反应过来了，oh, shit!

当然有个笨方法，查看版本历史，哪个文件改动了，一个一个把你改动的复制到bugfix分支。但是很傻，程序员不干体力劳动。

所以当然有更好的方案，git设计之初考虑到了这部分马虎的开发者。

stash
stash就是专门为马虎的开发者提供的功能，
大白话解释下什么事stash，stash可以把你当前版本的所有更改提取出来，放到储存区中，放心，这个时候你更改的代码不是消失了，只是换个地方保存起来。
切换回bugfix分支，再应用你的存储，这时候你在测试分支的修改bug的代码，就被带到bugfix分支了。
但需要注意的是，一定要记得每次新任务或临时做其他工作前提交一次代码，不一定非得提交到远程，本地一样可以。

顺便一提，远程分支的备注，一定要很标准，这样对大型项目的管理很有帮助
可以看下Vue项目yyx990803大神的提交备注。

xxx:xxxxxxxx

首先要有个大的前缀，标记你本地提交做的是哪方面的事，比如新加功能，修改bug，零碎的工作（无聊的工作）等等，接下来，描述本次更改具体完成的工作。如果只更改了一个文件，甚至可以把这个文件名放到：后面。
